# Episode 2 Script: The Rules of Engagement & Storage Safeguards
### Intro & The "Invisible" Storage Crisis

But right now, our workspace is an absolute blank slate. If we start building our private AMM infrastructure without an enforcement framework, our autonomous AI agent will default to messy, scattered, or non-portable configurations that clutter the host OS.
Remember the golden rule of this channel: **your typing days are over.** You should be pacing your living room, talking your architectural strategy out loud, and letting your AI co-pilot do the heavy lifting. But if you give an autonomous agent total freedom without boundaries, it becomes a loose cannon.
Today, we are shifting from a raw server to a professional, containerized development environment. We are going to establish the "Rules of Engagement"—a hidden configuration file that forces our AI to act as a disciplined, senior DevOps engineer instead of an overzealous intern.
But before we drop a single container into our environment, we have to handle the practical mechanics of our storage layout. By default, when you install Ollama on an Ubuntu machine, it writes its internal data and models directly to the primary operating system drive. For a high-frequency trading system that needs to run an intensely active XRPL validator node, this is a ticking time bomb. The heavy reads and writes from the AI engine will choke the primary drive's IOPS, cause the ledger node to lag behind consensus, and ultimately crash your trading loops.
To get that legendary, blazing-fast performance we talked about in Episode 1, our very first task is to cleanly migrate Ollama's model path permanently over to our secondary SSD array.
### Phase 1: Overriding the Systemd Trap
To change where a Linux system service writes its data, we have to inject an environment variable directly into its systemd background configuration. Now, there is a notorious trap here. Standard documentation will tell you to run sudo systemctl edit ollama. If you do that, Linux opens up a file packed with template comments, including a strict line that reads: *Lines below this comment will be discarded*. If you naturally paste your environment variables at the bottom of that file, the system silently deletes your settings the moment you hit save.
Getting everything back aligned after a path failure can be an absolute nightmare. To completely bypass this wrapper trap and save yourself hours of troubleshooting, we target the raw override configuration file directly in the terminal:
```bash
sudo mkdir -p /etc/systemd/system/ollama.service.d/
sudo nano /etc/systemd/system/ollama.service.d/override.conf

```
Inside this blank override file, we inject a clean, explicit service block. This single block locks down our cross-origin network broadcast from Episode 1 and explicitly dictates our secondary SSD path for all models, temporary caches, and context buffers:
```ini
[Service]
Environment="OLLAMA_HOST=0.0.0.0"
Environment="OLLAMA_ORIGINS=*"
Environment="OLLAMA_MODELS=/mnt/containers/ollama"

```
Once that is written to disk, we reload the system manager to register the changes, restart the core engine, and verify that the system successfully locked in our configuration:
```bash
sudo systemctl daemon-reload
sudo systemctl restart ollama
sudo systemctl show ollama --property=Environment

```
### Phase 2: Safely Migrating the Model Weights
Our configuration is now perfectly locked, but our existing model files are still sitting on the primary hard drive. We need to shift those heavy assets over to our secondary SSD to fully clear up our primary disk space.
Here is a massive architectural warning: *Never ask your AI agent to manage this specific step.* If you are pacing around your room and you casually ask Cline to handle this file migration, the first thing he will defensively do is stop the Ollama service to protect the files. The exact millisecond he kills that local service, he pulls his own plug, kills his connection to his brain, throws a connection refused error, and breaks your environment. As Rich found out.
Instead, we take control and execute the asset relocation directly in our standard terminal loop using a recursive copy that preserves all file attributes and permissions:
```bash
sudo cp -rp ~/.ollama/* /mnt/containers/ollama/

```
Once the files are completely duplicated onto the secondary SSD, we safely wipe the old data directory out of our primary home folder, freeing up immediate breathing room so our core OS and ledger node can run unhindered:
```bash
sudo rm -rf ~/.ollama

```
We let the system sit, let the storage stabilize, and boom—everything is perfectly aligned and running beautifully.
### Phase 3: Deploying the Container Engine
With our storage permanently optimized, we can now establish our foundational infrastructure rule: We refuse to clutter our bare-metal host with runtime dependencies, language environments, or loose database installations. Everything we build going forward—from our QuestDB time-series engine to our Java trading backend—must live inside isolated, highly portable containers.
To do this, we install the official Docker runtime engine directly to our server:
```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

```
To manage these containers fluidly without needing to prefix every single command with root privileges, we append our current user directly to the system's Docker security group:
```bash
sudo usermod -aG docker $USER

```
For Linux to actually evaluate and apply these new security group permissions to our active terminal profile, we must reset our connection state. Closing out the remote visual session and instantly re-establishing the SSH tunnel forces the server to recognize our new container management privileges.
### Phase 4: Initializing the Project Workspace
Now we initialize a clean, centralized home directory for our entire automated ecosystem. We create our project root directory, step directly into it, and open it as our primary visual workspace inside VS Code:
```bash
mkdir -p ~/xrpl-native-ecosystem && cd ~/xrpl-native-ecosystem

```
Opening this dedicated directory structures our file explorer sidebar, giving us a clean anchor point to build out our container blueprints and configuration rules.
### Phase 5: Enforcing AI Guardrails (.clinerules)
This brings us to our most powerful architectural tool: the .clinerules file. This is a hidden system configuration document placed directly in the root of our workspace.
Let me tell you why this file is non-negotiable. Without explicit guardrails, an autonomous coding agent can get overzealous. It can lose track of context and accidentally delete your core engineering files—it could literally wipe out your master blueprint.md file in the blink of an eye, leaving you praying that you have a recent backup copy to put back in place. Lucky Rich had a backup when he let Cline do cleanup.
To prevent catastrophes like that, the exact second our autonomous agent initializes inside this directory, he reads .clinerules. It completely overrides his default behavioral programming, strips away generic coding assumptions, and forces him to operate strictly under our defensive engineering standards and software constraints.
We create this file named exactly .clinerules in the project root and inject our explicit structural blueprint:
```text
# Custom System Instructions: XRPL Private AMM Trading Ecosystem

You are an expert software engineer specializing in Java, Next.js 14, and QuestDB.
You are acting as the primary assistant ("Cline") building a private algorithmic trading infrastructure.

## 1. System Core Architecture
- Ecosystem Nature: Private AMM pools on the XRPL.
- Infrastructure Constraint: EVERYTHING must be containerized.
All services (QuestDB, xrpld, Java Backend, Next.js Frontend) must run inside Docker containers orchestrated via docker-compose.yml.
Never install language runtimes directly on the host OS.
- Tech Stack: Java (StreamProcessor), QuestDB (Port 8812), Next.js 14, rippled/xrpld node (http://rippled:5005).

## 2. Hardcoded Software & Safety Constraints
- Trade Sizing: Use dynamic stack allocation: TRADE_SIZE_PERCENTAGE = 0.20 (20% of available balance).
- Liquidity Guardrail: MINIMUM_POOL_DEPTH_XRP = 100.0. If liquidity is less, abort and log "Liquidity Guardrail Skip."

## 3. Coding Standards
- Provide production-ready, defensively engineered code.
- Always implement graceful error handling for QuestDB and JSON-RPC connection loops.

```
