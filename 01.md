### Phase 1: Designing the Dual-Drive Layout
The very first architectural step to ensuring system stability is mapping our hardware storage strategy. We are intentionally separating our server into two distinct physical NVMe arrays for a critical performance reason.
The primary drive, mounted to the root directory, is dedicated **exclusively** to the Operating System and our live XRPL node. The XRP Ledger writes transactional data to disk continuously using NuDB, pulling thousands of random disk IOPS to maintain consensus synchronization with the network. It requires unhindered, low-latency access to disk throughput and must own this physical drive completely.
Our secondary drive, mounted to /mnt/containers, is dedicated to everything else. This includes our local Ollama research caches, our QuestDB time-series backend, and all Docker container execution files. By cleanly offloading our background applications, model weights, and temporary database caches to this secondary SSD, we guarantee that our high-frequency trading loops and ledger synchronization never experience disk I/O starvation. When you isolate the primary drive for Ripple and offload the rest, the response times when you ask your agent to execute are **blazing fast**.
### Phase 2: Securing the Handshake (SSH Key Exchange)
With our storage architecture planned, we need to lock down our communications. We refuse to leave password vulnerability active on our server; instead, we switch to high-security cryptographic keys.
On our local laptop's terminal, we generate an ultra-secure Ed25519 key pair:
```bash
ssh-keygen -t ed25519 -C "your_email@example.com"

```
Once generated, we push our public identity token across the network directly to the Ubuntu machine:
```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server_ip

```
Now that our public key is safely authorized, we SSH into the server to harden the system's background daemon so hackers can't brute-force our logins:
```bash
sudo nano /etc/ssh/sshd_config

```
Inside this configuration file, we locate PubkeyAuthentication and explicitly set it to yes. We then find PasswordAuthentication and drop it to no. We save the file using Ctrl+O, hit Enter, and exit with Ctrl+X. To instantly apply the cryptographic lockdown, we restart the background service:
```bash
sudo systemctl restart sshd

```

### Phase 4: Booting the Research Engine (Ollama & Open WebUI)
With our coding agent live, we now deploy our secondary brain: our interactive research engine. We start by running the official open-source installation script directly on the server to ignite Ollama:
```bash
curl -fsSL https://ollama.com/install.sh | sh

```
By default, the Ollama engine only listens to internal communication loops inside the server. Since we want to control this engine across our local network via a beautiful web dashboard, we open up its background service parameters:
```bash
sudo systemctl edit ollama.service

```
Inside the systemd template file, we inject two environment overrides to broadcast across our local network and accept cross-origin requests:
```ini
[Service]
Environment="OLLAMA_HOST=0.0.0.0"
Environment="OLLAMA_ORIGINS=*"

```
We save and close out of the editor. To allow this traffic through Ubuntu's active network security layer, we punch a hole for the API port, reload our system configurations, and restart the background engine:
```bash
sudo ufw allow 11434/tcp
sudo systemctl daemon-reload
sudo systemctl restart ollama

```
Because we have isolated this Ollama instance entirely onto our secondary drive array, it can process local models and manage deep research data without ever bottlenecking the primary drive.
